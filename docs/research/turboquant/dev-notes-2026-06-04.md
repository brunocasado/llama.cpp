## What was added in phase 2
- public enum `llama_kv_cache_codec_type` in `include/llama.h`
- public helper `llama_kv_cache_codec_type_name()`
- `llama_context_params.kv_cache_codec_type`
- common CLI/config plumbing via `--kv-cache-codec`
- `turboquant` codec stub selectable from the CLI
- behavior still defaults to `legacy`

## What was added in phase 3 (test-oriented)
- `llama_model::create_memory()` now recognizes codec capability and can force quantized KV storage for the `turboquant` codec stub
- current behavior for `turboquant` is intentionally simple and measurable:
  - it forces `type_k = q4_0`
  - it forces `type_v = q4_0`
- `llama-bench` now accepts `--kv-cache-codec`
- `llama-bench` records `kv_cache_codec` in structured output

This is not the final TurboQuant algorithm, but it is the first point where selecting the codec can produce a real, user-visible change in the runtime configuration.

## Why this seam is the right one
This follows upstream structure instead of fighting it:
- `common/arg.cpp` parses user intent
- `common/common.cpp` maps user params to `llama_context_params`
- `llama_context` builds `llama_memory_params`
- `llama_model::create_memory()` decides which memory implementation to instantiate
- `llama_kv_cache*` and hybrid/recurrent memory implementations remain mostly untouched so far

This keeps legacy behavior identical while opening a clean slot for a future real TurboQuant codec.

With a normal GGUF model, compare:
```bash
./build/bin/llama-bench -m /path/to/model.gguf -p 1024 -n 128 -r 3 -ctk f16 -ctv f16 --kv-cache-codec legacy -o jsonl
./build/bin/llama-bench -m /path/to/model.gguf -p 1024 -n 128 -r 3 -ctk f16 -ctv f16 --kv-cache-codec turboquant -o jsonl
```

At the current stage, the second command forces quantized KV storage (`q4_0/q4_0`) through the codec path, so you may already observe a memory/perf difference. It is still a proxy for the real TurboQuant path, not the final algorithm.
